# Requirements Document

## Introduction

This document defines the requirements for the production-grade onboarding flow of a multi-tenant SaaS CRM system. The onboarding flow covers the complete lifecycle from initial Owner account creation through organization setup to the invitation-based onboarding of subsequent team members. The system is built on an Express/TypeScript/Prisma backend and a Next.js 16 App Router frontend with RBAC already in place. The current implementation is partially built; this spec formalizes the expected behavior, fills gaps, and ensures correctness and security for all onboarding paths.

---

## Glossary

- **Owner**: The first user who registers. Creates and owns the organization. There is exactly one Owner per organization. The Owner holds the `owner` role.
- **Admin**: A team member with elevated CRM management access, invited by an Owner.
- **Sales_Manager**: A team member who manages deals and contacts, invited by an Owner or Admin.
- **Sales_Rep**: A team member who manages their own leads, invited by an Owner, Admin, or Sales_Manager.
- **Marketing**: A team member with contact import and analytics access, invited by an Owner or Admin.
- **Support**: A team member with read-only CRM access, invited by an Owner or Admin.
- **Invited_User**: Any non-Owner user who joins via an invitation link. Includes Admin, Sales_Manager, Sales_Rep, Marketing, and Support roles.
- **Organization**: A tenant in the multi-tenant system. Has exactly one Owner. Created atomically during Owner registration with a placeholder state (`setupComplete = false`) until the Owner completes the Setup Wizard.
- **Setup_Wizard**: A multi-step frontend form at `/onboarding/setup` that collects Organization profile data. Only accessible to an authenticated Owner whose Organization has `setupComplete = false`.
- **Invitation**: A time-limited (72-hour) email token sent to a recipient's email address granting them the ability to create a user account and join the Organization.
- **JWT**: A JSON Web Token issued upon successful authentication, carrying `userId`, `organizationId`, `roleId`, and `roleName`.
- **Auth_Service**: The backend service handling login, registration, and profile management (`/auth/*` routes).
- **Onboarding_Service**: The backend service handling Owner registration and Organization setup completion.
- **Invitation_Service**: The backend service handling invitation sending, acceptance, resending, and revocation.
- **Route_Guard**: The frontend mechanism (in `RootLayoutClient`) that enforces authentication and setup completion before granting access to protected routes.
- **Org_ID**: A UUID generated by the system and assigned to an Organization at creation time.
- **Token_Hash**: A bcrypt hash of the raw invitation token stored in the database. The raw token travels only in the invitation email link.
- **setupComplete**: A boolean field on the Organization model. When `false`, the Owner is redirected to the Setup_Wizard on every authenticated page load. When `true`, the Owner may access the dashboard.

## Requirements

### Requirement 1: Owner Account Registration

**User Story:** As a new Owner, I want to create my personal account using a username, email, and password, so that I can gain a JWT and proceed to configure my organization.

#### Acceptance Criteria

1. WHEN a `POST /auth/register` request is received with a valid `name`, `email`, and `password`, THE Onboarding_Service SHALL create an Owner User record and an Organization shell atomically within a single database transaction.
2. THE Onboarding_Service SHALL set `setupComplete = false` on the newly created Organization.
3. THE Onboarding_Service SHALL seed all system Role records and Permission records for the Organization within the same registration transaction.
4. THE Onboarding_Service SHALL assign the Owner User the `owner` role and set `isOwner = true` on the User record.
5. WHEN registration completes successfully, THE Auth_Service SHALL return a JWT containing `userId`, `organizationId`, `roleId`, and `roleName` for the Owner.
6. THE registration response SHALL include a `requiresSetup: true` field so the frontend can redirect to the Setup_Wizard without an additional network request.
7. IF a `POST /auth/register` request is received with an email address that already exists in the database, THEN THE Onboarding_Service SHALL return an HTTP 400 error with the message "An account with this email already exists"; this check SHALL take precedence over any concurrent validation failures.
8. IF the `password` field is shorter than 8 characters, THEN THE Auth_Service SHALL return an HTTP 422 validation error before processing the request.
9. IF the `name` field is empty or missing, THEN THE Auth_Service SHALL return an HTTP 422 validation error before processing the request.
10. THE Onboarding_Service SHALL hash the Owner's password using bcrypt with a cost factor of 12 before persisting it.
11. WHEN registration completes successfully, THE Onboarding_Service SHALL write an `AuditLog` entry with `action = 'create'`, `resource = 'organization'`, and `metadata.step = 'owner_registered'`.

### Requirement 2: Post-Registration Redirect to Setup Wizard

**User Story:** As a newly registered Owner, I want to be automatically redirected to the Organization Setup Wizard immediately after account creation, so that I cannot access the CRM dashboard with an unconfigured organization.

#### Acceptance Criteria

1. WHEN the frontend registration form is submitted successfully and the response contains `requiresSetup: true`, THE Registration_Form SHALL redirect the Owner to `/onboarding/setup`.
2. WHEN an authenticated Owner whose Organization has `setupComplete = false` navigates to any protected route other than `/onboarding/setup`, THE Route_Guard SHALL redirect the Owner to `/onboarding/setup`.
3. WHILE an Owner's Organization has `setupComplete = false`, THE Route_Guard SHALL prevent the Owner from accessing `/dashboard` or any other non-setup protected route.
4. WHEN an authenticated Owner whose Organization has `setupComplete = true` navigates to `/onboarding/setup`, THE Route_Guard SHALL redirect the Owner to `/dashboard`.
5. WHEN an authenticated Owner whose Organization has `setupComplete = true` navigates to `/onboarding/setup`, THE Setup_Wizard page SHALL redirect to `/dashboard` without rendering the wizard; unauthenticated users SHALL be redirected to `/login` before any setup-completion redirect is evaluated.

---

### Requirement 3: First Login Detection and Routing

**User Story:** As an Owner who has registered but not yet completed setup, I want to be automatically redirected to the Setup Wizard on my first login, so that the system always routes me correctly based on my setup state.

#### Acceptance Criteria

1. WHEN a `POST /auth/login` request succeeds for a user, THE Auth_Service SHALL return the full user object including the associated Organization record so the frontend can evaluate `setupComplete`.
2. WHEN the login response is received and the Organization has `setupComplete = false`, THE Login_Page SHALL redirect the authenticated user to `/onboarding/setup`.
3. WHEN the login response is received and the Organization has `setupComplete = true`, THE Login_Page SHALL redirect the authenticated user to `/dashboard`.
4. WHEN an already-authenticated user navigates to `/login`, THE Route_Guard SHALL redirect the user to `/dashboard` if `setupComplete = true`, or to `/onboarding/setup` if `setupComplete = false`.
5. THE Auth_Service SHALL update the `lastLoginAt` timestamp on the User record for every successful login.

### Requirement 4: Organization Setup Wizard

**User Story:** As an authenticated Owner with an incomplete organization, I want to complete a multi-step Setup Wizard to configure my organization's profile, so that my CRM workspace is properly configured before I begin using it.

#### Acceptance Criteria

1. WHEN the Setup_Wizard is submitted via `POST /auth/setup` with required fields (`name`, `industry`, `companySize`, `country`, `currency`, `timezone`), THE Onboarding_Service SHALL update the Organization record and set `setupComplete = true`.
2. THE Setup_Wizard SHALL collect the following required fields: Organization Name, Industry, Company Size, Country, Default Currency, and Default Timezone.
3. THE Setup_Wizard SHALL accept the following optional fields: Company Logo URL, Business Address, Phone Number, Website URL, and Fiscal Year Start Month.
4. IF the `name` field is empty in the setup request, THEN THE Onboarding_Service SHALL return an HTTP 422 validation error and SHALL NOT update the Organization record.
5. IF the `industry` field is missing in the setup request, THEN THE Onboarding_Service SHALL return an HTTP 422 validation error.
6. IF the `companySize` field is missing in the setup request, THEN THE Onboarding_Service SHALL return an HTTP 422 validation error.
7. WHEN a country is selected in the Setup_Wizard, THE Setup_Wizard SHALL auto-suggest the default currency and timezone for that country while allowing the Owner to override both values; no suggestion SHALL be made until a country has been selected.
8. WHEN the setup form is submitted successfully, THE Onboarding_Service SHALL regenerate the Organization slug from the submitted `name` field to replace the placeholder slug created at registration.
9. WHEN the setup is completed, THE Onboarding_Service SHALL write an `AuditLog` entry with `action = 'update'`, `resource = 'organization'`, and `metadata.step = 'setup_complete'`.
10. WHEN `POST /auth/setup` is called, THE Auth_Service middleware SHALL verify the JWT and ensure the caller has a valid `organizationId` before the Onboarding_Service processes the request.
11. WHEN setup completes successfully, THE Setup_Wizard SHALL redirect the Owner to `/dashboard`.
12. WHEN `POST /auth/setup` is called and the Organization already has `setupComplete = true`, THE Onboarding_Service SHALL update the organization settings with the provided data, SHALL set `setupComplete = true`, and SHALL return a success response without re-running field validation for that call.

---

### Requirement 5: Owner Dashboard Access

**User Story:** As an Owner who has completed organization setup, I want to access my dashboard to manage users, settings, and CRM data, so that I can begin operating my workspace.

#### Acceptance Criteria

1. WHEN an authenticated Owner navigates to `/dashboard` and the Organization has `setupComplete = true`, THE Route_Guard SHALL allow access to the dashboard.
2. THE dashboard SHALL be accessible only to authenticated users whose Organization has `setupComplete = true`.
3. WHILE a user is authenticated with the `owner` role, THE dashboard SHALL display team management capabilities including the ability to send invitations.
4. WHEN an Owner accesses the dashboard, THE Auth_Service SHALL return the Organization name, country, and currency via `GET /auth/me` for display in the UI.

---

### Requirement 6: Invitation Sending

**User Story:** As an Owner or Admin, I want to send email invitations to new team members with a specific role, so that they can join my organization.

#### Acceptance Criteria

1. WHEN a `POST /invitations` request is received from an authenticated user with the `user.invite` permission, THE Invitation_Service SHALL create an Invitation record with `status = 'pending'` and a 72-hour expiry.
2. THE Invitation_Service SHALL generate a cryptographically secure random token of 32 bytes (64 hex characters) for each invitation.
3. THE Invitation_Service SHALL store only the bcrypt hash (cost factor 10) of the raw token in the database.
4. THE Invitation_Service SHALL return the raw token to the caller so it can be embedded in the invitation email link.
5. IF a pending invitation already exists for the same email address within the same Organization, THEN THE Invitation_Service SHALL expire the existing invitation and create a new one.
6. IF the target role does not exist within the caller's Organization, THEN THE Invitation_Service SHALL return an HTTP 404 error.
7. IF the caller's role is not permitted to invite the target role according to the invitation hierarchy, THEN THE Invitation_Service SHALL return an HTTP 403 error with a descriptive message.
8. IF the email address belongs to a user who already has an active account in the Organization, THEN THE Invitation_Service SHALL return an HTTP 400 error.
9. THE invitation hierarchy SHALL enforce: Owner can invite Admin, Sales_Manager, Sales_Rep, Marketing, Support, and other Owners; Admin can invite Sales_Manager, Sales_Rep, Marketing, Support; Sales_Manager can invite Sales_Rep; Sales_Rep, Marketing, and Support cannot invite anyone.
10. WHEN an invitation is created, THE Invitation_Service SHALL write an `AuditLog` entry with `action = 'invite_sent'` and `metadata` containing the invited email and target role name.

---

### Requirement 7: Invitation Email Delivery

**User Story:** As a system, I want to deliver invitation emails containing a secure one-time link to invited users, so that they can accept the invitation and join the organization.

#### Acceptance Criteria

1. WHEN an invitation is created successfully, THE system SHALL send an email to the invited address containing a link of the form `{FRONTEND_URL}/invitations/accept?token={rawToken}`.
2. THE invitation email SHALL include the inviting Organization's name and the inviter's name so the recipient understands the context of the invitation.
3. THE invitation email SHALL state the expiration deadline (72 hours from the time of sending).
4. IF email delivery fails, THEN THE system SHALL log the failure and SHALL NOT roll back the Invitation record; the invitation SHALL remain valid and can be resent.
5. WHERE an email service is not yet configured, THE system SHALL log the raw invitation token at INFO level to allow manual testing without rolling back the invitation creation.

---

### Requirement 8: Invitation Acceptance

**User Story:** As an Invited_User who has received an invitation email, I want to accept the invitation by creating my username and password, so that I can join the organization.

#### Acceptance Criteria

1. WHEN a `POST /invitations/accept` request is received with a valid `token`, `name`, and `password`, THE Invitation_Service SHALL verify the token by bcrypt-comparing it against all pending invitations and SHALL only create a User record when the token is valid, not expired, and the email address does not already have an active User account in the Organization.
2. IF the token does not match any pending invitation, THEN THE Invitation_Service SHALL return an HTTP 400 error with the message "Invalid or expired invitation token".
3. IF the matched invitation has an `expiresAt` timestamp in the past, THEN THE Invitation_Service SHALL set `status = 'expired'` on the invitation and return an HTTP 400 error with the message "This invitation has expired. Please request a new one."
4. WHEN a valid invitation is accepted, THE Invitation_Service SHALL create a new User record within the Organization assigned to the invited role, within an atomic database transaction.
5. WHEN a valid invitation is accepted, THE Invitation_Service SHALL set `status = 'accepted'` and `acceptedAt = now()` on the Invitation record within the same transaction.
6. WHEN acceptance is complete, THE Invitation_Service SHALL return a JWT for the new User containing `userId`, `organizationId`, `roleId`, and `roleName`.
7. WHEN acceptance is complete, THE Invitation_Service SHALL write an `AuditLog` entry with `action = 'invite_accepted'`.
8. THE `POST /invitations/accept` endpoint SHALL be public (no JWT required) because the invitation token itself serves as the credential.
9. IF the invited email address already has an active User account in the Organization, THEN THE Invitation_Service SHALL return an HTTP 400 error.
10. THE Invitation_Service SHALL hash the new User's password using bcrypt with a cost factor of 12 before persisting it.
11. WHEN the invitation acceptance page loads, THE Accept_Invitation_Page SHALL read the `token` query parameter from the URL and display an error state if the token is absent.

---

### Requirement 9: Post-Acceptance Profile Completion

**User Story:** As an Invited_User who has just accepted an invitation, I want to complete my personal profile (display name, job title, and profile picture) on first login, so that my teammates can identify me.

#### Acceptance Criteria

1. WHEN an Invited_User authenticates for the first time after accepting an invitation and the User record has no `jobTitle` or `avatar` beyond the auto-generated initials, THE frontend SHALL present a profile completion prompt.
2. THE profile completion form SHALL collect: display name (pre-filled from the acceptance form), job title (optional), and profile picture URL or upload (optional).
3. WHEN profile data is submitted, THE Auth_Service SHALL update the User record via `PATCH /auth/me` with the provided `name` and `avatar` fields.
4. IF the profile completion step is skipped, THE system SHALL allow the Invited_User to proceed to the dashboard; profile completion SHALL NOT block CRM access.
5. WHEN an Invited_User logs in, THE Route_Guard SHALL NOT redirect the user to `/onboarding/setup` because the Organization is already configured; the user SHALL be routed to `/dashboard`, and IF the dashboard redirect fails or is blocked, THE Route_Guard SHALL fall back to displaying the login page.

---

### Requirement 10: Invited User Organization Association

**User Story:** As the system, I want to automatically associate every Invited_User with the inviting organization, so that users never need to select or create an organization.

#### Acceptance Criteria

1. WHEN an Invited_User accepts an invitation, THE Invitation_Service SHALL assign `organizationId` from the Invitation record to the new User record.
2. THE Invitation_Service SHALL assign the `roleId` from the Invitation record to the new User record without allowing the Invited_User to choose a different role.
3. THE system SHALL NOT present any organization creation UI to Invited_Users at any point in the acceptance or login flow.
4. IF an authenticated Invited_User attempts to call `POST /auth/setup`, THEN THE Onboarding_Service SHALL verify that the caller has the `owner` role and return an HTTP 403 error if the caller is not an Owner.
5. THE JWT issued to an Invited_User upon acceptance SHALL contain the correct `organizationId` so that all subsequent API requests are automatically scoped to the correct tenant.

---

### Requirement 11: Invitation Management

**User Story:** As an Owner or Admin, I want to list, resend, and revoke pending invitations, so that I can manage access to my organization.

#### Acceptance Criteria

1. WHEN `GET /invitations` is called by an authenticated user with the `user.invite` permission, THE Invitation_Service SHALL return all pending invitations for the caller's Organization ordered by `createdAt` descending.
2. WHEN `POST /invitations/:id/resend` is called, THE Invitation_Service SHALL expire the existing invitation, create a new invitation with a fresh 32-byte token and a new 72-hour expiry, and return the new raw token.
3. WHEN `DELETE /invitations/:id` is called, THE Invitation_Service SHALL set `status = 'revoked'` on the target invitation.
4. IF the invitation to be resent or revoked does not belong to the caller's Organization, THEN THE Invitation_Service SHALL return an HTTP 404 error.
5. IF the caller's role is not permitted to invite the role of the invitation being resent, THEN THE Invitation_Service SHALL return an HTTP 403 error.
6. THE `GET /invitations`, `POST /invitations/:id/resend`, and `DELETE /invitations/:id` endpoints SHALL require authentication and the `user.invite` permission.

---

### Requirement 12: Organization Uniqueness and Ownership Invariants

**User Story:** As the system, I want to enforce that each Organization has exactly one Owner and that only Owners can create Organizations, so that multi-tenant data isolation is maintained.

#### Acceptance Criteria

1. THE system SHALL ensure that each Organization has exactly one User with `isOwner = true` at all times.
2. THE system SHALL ensure that an Owner cannot be demoted from the `owner` role by any API action without transferring ownership to another user first.
3. IF a non-Owner authenticated user calls `POST /auth/register` for a second account, THE system behavior SHALL be that the registration creates a second Owner and a second Organization — this is acceptable only because registration creates a brand new tenant. A user with an existing account SHALL NOT be able to register a new Owner account using their existing email.
4. THE Onboarding_Service SHALL enforce that the `POST /auth/setup` endpoint can only be called by a user whose JWT `roleName` is `'owner'`; any other role SHALL receive an HTTP 403 response. IF the authorization check fails for any reason other than an invalid JWT, THE Onboarding_Service SHALL log the unauthorized attempt and return HTTP 403 without proceeding.
5. THE system SHALL generate a unique Org_ID (UUID) for every Organization at creation time and SHALL NOT allow external callers to specify the Org_ID. IF UUID generation fails, THE Onboarding_Service SHALL reject the Organization creation request with an HTTP 500 error until generation is restored.

---

### Requirement 13: Security and Token Integrity

**User Story:** As the system, I want to ensure all authentication tokens and invitation tokens are cryptographically secure and cannot be forged or replayed, so that the system resists unauthorized access.

#### Acceptance Criteria

1. THE Auth_Service SHALL sign JWTs using a secret key loaded from environment configuration and SHALL NOT use a hardcoded signing key.
2. THE Invitation_Service SHALL generate invitation tokens using `crypto.randomBytes(32)` and SHALL NOT use predictable or sequential values.
3. THE Invitation_Service SHALL store only the bcrypt hash of the invitation token in the database so that a database breach does not expose usable tokens.
4. WHEN an invitation token is used to accept an invitation, THE system SHALL compare the submitted raw token against the stored hash using constant-time bcrypt comparison to prevent timing attacks.
5. IF a JWT is missing, malformed, expired, or fails signature verification for any reason, THEN THE Auth_Service middleware SHALL return an HTTP 401 error before executing any protected handler.
6. THE system SHALL enforce HTTPS-only token transmission in production; invitation links SHALL use the configured `FRONTEND_URL` environment variable as the base URL.
7. WHEN a password is submitted (registration, invitation acceptance, or password change), THE system SHALL reject passwords shorter than 8 characters before hashing.

---

### Requirement 14: Audit Logging

**User Story:** As an Owner or Admin, I want all onboarding and authentication actions to be recorded in the audit log, so that I can review who joined the organization and when.

#### Acceptance Criteria

1. THE system SHALL create an `AuditLog` entry for each of the following events: Owner registration, Organization setup completion, user login, invitation sent, invitation accepted, invitation revoked.
2. EACH `AuditLog` entry SHALL include `organizationId`, `userId`, `action`, `resource`, `resourceId`, and a `metadata` JSON field with event-specific context.
3. THE `AuditLog` entries for invitation events SHALL record the invited email address and assigned role name in the `metadata` field.
4. WHERE the `audit.view` permission is granted to a role, THE system SHALL allow users with that role to read the AuditLog for their Organization via the existing audit routes.
5. THE `AuditLog` table SHALL use `onDelete: SetNull` for the `userId` foreign key so that deleting a user does not cascade-delete audit history.
